Gamut checking and mapping for sRGB color space.

Not all LAB colors can be represented in sRGB! Some theoretical colors that humans can perceive simply can‚Äôt be displayed on a monitor. This module helps you: 1. Check if a LAB color is displayable (in gamut) 2. Find the nearest displayable color if it‚Äôs not

The key insight: sRGB is a cube (R, G, B all 0-255), but LAB describes a much larger volume. The intersection is the ‚ÄúsRGB gamut.‚Äù

color_tools.gamut.is_in_srgb_gamut(lab, tolerance=None)[source]ÔÉÅ

Check if a LAB color can be represented in sRGB without clipping.

This is the fundamental question of color reproduction: ‚ÄúCan my monitor actually show this color, or will it have to fake it by clamping?‚Äù

How it works: Convert LAB ‚Üí RGB without clamping. If all RGB components are naturally in the 0-255 range (with a small tolerance), the color is in gamut!

Why tolerance? Floating point math is imprecise. A value of 255.0000001 should be considered ‚Äúin gamut‚Äù even though it‚Äôs technically over 255.

Examples of out-of-gamut colors: - Super saturated colors at mid-lightness (like ‚Äúlaser red‚Äù) - Colors that would need negative RGB values - Colors that would need RGB values > 255

Parameters:

  • lab (Tuple[float, float, float]) ‚Äì L*a*b* color tuple

  • tolerance (float | None) ‚Äì How close to 0/255 boundaries before considering out-of-gamut. If None, uses the global config value (default 0.01).

Return type:

bool

Returns:

True if the color can be represented in sRGB, False if it would clip

Example

>>> is_in_srgb_gamut((50, 0, 0))  # Mid gray
True
>>> is_in_srgb_gamut((50, 150, 100))  # Super saturated - impossible!
False
color_tools.gamut.find_nearest_in_gamut(lab, max_iterations=None)[source]ÔÉÅ

Find the nearest in-gamut LAB color to a given (possibly out-of-gamut) LAB color.

The Strategy: When a color is out of gamut, it‚Äôs usually because it‚Äôs too saturated (high chroma). So we convert to LCH, then use binary search to find the maximum chroma that still fits in sRGB. This preserves: - ‚úÖ Lightness (L) - stays the same - ‚úÖ Hue (h) - stays the same - ‚ùå Chroma (C) - reduced until it fits

Why this works: Think of it like turning down the ‚Äúsaturation‚Äù slider in Photoshop until the color becomes displayable. The hue doesn‚Äôt change (red stays red), it just becomes less ‚Äúpunchy.‚Äù

Binary search magic: Instead of trying every possible chroma value (0.0, 0.01, 0.02, ‚Ä¶), we do a binary search. Each iteration cuts the search space in half. After 20 iterations, we‚Äôve checked 2^20 = 1 million possible values! üöÄ

Parameters:

  • lab (Tuple[float, float, float]) ‚Äì Potentially out-of-gamut L*a*b* color

  • max_iterations (int | None) ‚Äì Maximum desaturation steps (binary search iterations). If None, uses global config value (default 20).

Return type:

Tuple[float, float, float]

Returns:

In-gamut L*a*b* color (closest match that can be displayed)

Example

>>> # Super saturated red - can't be displayed!
>>> out_of_gamut = (50, 100, 80)
>>> displayable = find_nearest_in_gamut(out_of_gamut)
>>> # Result: same lightness and hue, but less saturated
>>> is_in_srgb_gamut(displayable)
True
color_tools.gamut.clamp_to_gamut(lab)[source]ÔÉÅ

Convert LAB to RGB and back to LAB (with clamping).

This is the ‚Äúquick and dirty‚Äù approach to gamut mapping. Instead of carefully desaturating, we just let lab_to_rgb() clamp the RGB values to 0-255, then convert back to LAB to see what we got.

Pros: Fast!

Cons: Can change hue (not just chroma). A pure red might become a slightly orange-red after clamping.

For most applications, find_nearest_in_gamut() is better because it preserves hue. This function exists for when you need speed over accuracy.

Parameters:

lab (Tuple[float, float, float]) ‚Äì Any L*a*b* color (in-gamut or not)

Return type:

Tuple[float, float, float]

Returns:

In-gamut L*a*b* color (clamped)
